# analysis_prompt_v3.md 設計書

## Context

ユーザーは週末に `normalized_data/normalized_data_YYYY-MM-DD_to_YYYY-MM-DD.json` を入力として、週次学習分析プロンプトを実行している。現在のプロンプト（`analysis_prompt_v2.md`）は正規化JSONデータに対応しているが、以下の理由で最新版への更新が必要になった：

1. **プロジェクトの学習メソッドが進化**: Cognitive Headroom、Focus Metric、Friction、Satisfactionなどの新メトリクスが導入され、より精緻な分析が可能になった
2. **Daily Review v2フォーマットの確立**: Stats 11フィールド、AI生成フィードバック（Top1/Done条件達成度、時間配分フィードバック）が標準化され、週次分析での活用が期待される
3. **週次戦略フレームワークの体系化**: The One Thing、Success Criteria、Risk Management（If-Thenプラン）が確立し、週次分析との連携が明確になった
4. **時間帯別学習特性の発見**: 午前ブロック（DS平均4.2）、夜間ブロック（Headroom=2、認知疲弊）などのパターンが蓄積され、分析に組み込むべきデータが揃った

この設計書は、これらの最新要素を統合した `analysis_prompt_v3.md` の実装指針を定める。

---

## 1. 現在のv2の問題点

### 1.1 最新メトリクスへの未対応
- **問題**: Deep Score以外のメトリクス（Cognitive Headroom、Focus Metric、Friction、Satisfaction）が分析項目に含まれていない
- **影響**: 集中度だけでなく認知リソースや心理的摩擦の分析ができず、阻害要因の特定が不十分

### 1.2 Daily Review v2フォーマットの活用不足
- **問題**: Stats 11フィールド（Budget、切れたら→など）やAI生成フィードバック（定量面+定性面）への言及がない
- **影響**: 日次レベルで既に分析されたTop1達成度や時間配分の洞察を週次分析で活用できていない

### 1.3 時間帯別分析の視点が弱い
- **問題**: 「午前・午後・夜間ブロック」という時間帯別の学習特性が分析フレームワークに組み込まれていない
- **影響**: 時間帯によるDS平均の差異（午前4.2、夜間2.8など）や認知リソースの変化を見逃す

### 1.4 週次戦略フレームワークとの連携不明確
- **問題**: The One Thing実験、Success Criteria、Risk Managementなどの概念がBuild/Pivotセクションで明示されていない
- **影響**: 分析結果が次段階プロンプト（`generate_weekly_strategy_from_chat.md`）への入力として機能しにくい

### 1.5 出力フォーマットの具体性不足
- **問題**: 各セクションの出力例がなく、「スマホで読める分量」という指示も曖昧
- **影響**: AI出力の品質が安定せず、毎回異なる粒度・構造で出力される可能性

## 2. v3での改善内容

### 2.1 入力データ定義の強化
- **Daily Review v2の最新フィールドを明示**: Stats 11項目（Day Mode、Budget、Total Min、純粋な学習時間、休憩時間、運動時間、Avg Deep Score、Top1、Done条件、切れたら→）を入力データ説明に追加
- **AI生成フィードバックの活用を明記**: `■ Top1 / Done条件 達成度フィードバック`（定量面+定性面）と`■ 時間配分フィードバック`（定量面+定性面）を分析の基礎データとして位置づけ
- **normalized_data JSONと元Daily MDファイルの参照関係を記載**: 週次分析ではJSONデータを使うが、詳細な引用は元のDaily MDファイルから行うことを明示

### 2.2 分析項目の拡張

#### Measureセクション
- **時間帯別分析を追加**: 午前ブロック（07:00-12:00）、午後ブロック（12:00-17:00）、夜間ブロック（17:00-22:00）のDS平均とCognitive Headroom推定
- **新メトリクスの分析を追加**: Satisfaction（満足度）、Focus Metric（集中持続度）、Friction（摩擦度）の週次トレンド
- **Budget超過の解釈を精緻化**: Flow状態継続による肯定的超過 vs 計画甘さによる超過を区別

#### Learnセクション
- **Flow状態（DS 4.0以上）の維持パターン分析を追加**: Flow中の「切り替えない」判断の成功例を抽出
- **週次戦略experiment検証を追加**: `experiment.action`が実際に実行されたか、`hypothesis`が正しかったかをデータで検証
- **認知リソース管理の分析を追加**: Cognitive Headroom低下時のタスク選択（切れたら→の活用）が適切だったか

#### Build/Pivotセクション
- **The One Thing提案を明示**: 来週試すべき1つの改善を具体的に提案
- **Risk Management（If-Thenプラン）を追加**: 想定されるリスクと対策を事前設計
- **Success Criteriaを追加**: 来週の成功を判断する具体的基準を設定

### 2.3 出力フォーマットの具体化
- **各セクションの出力例を提示**: Measure、Learn、Build/Pivotそれぞれの記述例を示す
- **引用形式の詳細化**: 日付参照`[YYYY-MM-DD]`、セッション引用、Daily Review引用、AI生成フィードバック引用の5つのルールを明記
- **出力長さの目安を設定**: 全体2500-3500文字（A4用紙2-3枚相当）、Measure 800-1200文字、Learn 600-1000文字、Build/Pivot 400-600文字

### 2.4 プロンプト構造の改善
- **役割定義の明確化**: 「熟練のアジャイル・ラーニング・コーチ」として、週次振り返りで「内省（Reflection）」を支援する役割を明示
- **分析視点・質問構造の具体化**: 各セクションで問うべき質問を具体的に記述（例：「Flow状態が続いた日に共通する条件は？」）
- **トーン設定の詳細化**: 「客観的かつコーチトーン」を具体化し、「データから導かれる気づき」を提示する姿勢を明示

### 2.5 次段階プロンプトへの連携
- **`generate_weekly_strategy_from_chat.md`への入力フォーマットを意識**: Build/Pivotセクションの出力が次段階プロンプトの「3. 重点実験」「4. 想定されるリスク」に直接活用できる形式に
- **`gap_analysis_prompt.md`との役割分担を明確化**: このプロンプトは「実績データの内省」に特化し、ギャップ分析は別プロンプトで行うことを明記
- **オプションJSON出力を追加**: 機械処理用に構造化データ（`analysis_summary`、`key_patterns`、`proposed_experiment`、`proposed_risks`）を出力可能に

## 3. プロンプト構成案

### 3.1 役割定義

```markdown
あなたは「熟練のアジャイル・ラーニング・コーチ」です。

提供された「週次学習ログ（正規化済みJSONデータ）」を分析し、ユーザーの「内省（Reflection）」を支援するレポートを作成してください。

**このプロンプトの役割**:
- 1週間の学習実績を定量・定性の両面から分析
- 勝ちパターン（Good & Pattern）とボトルネック（Bottle Neck）を抽出
- 来週の改善実験（The One Thing）とリスク対策を提案

前段階の正規化フェーズで構造化されたJSONデータと、各日のDaily Review（v2フォーマット）に含まれるAI生成フィードバックを活用し、**「行動の質・感情・環境要因・戦略」**に焦点を当てた定性的分析を行います。
```

### 3.2 入力データ形式

```markdown
【入力データ形式】

提供されるデータは週次分析用に正規化されたJSONです。

#### 1. JSON構造（`normalized_data/normalized_data_YYYY-MM-DD_to_YYYY-MM-DD.json`）
- `period`: 分析対象期間（`start_date`, `end_date`）
- `weekly_summary`: 週全体の集計値
  - `total_hours`: 週の合計学習時間（時間単位）
  - `average_deep_score`: 週平均Deep Score
- `days`: 日ごとの詳細データ（配列）
  - `date`: 日付（YYYY-MM-DD）
  - `day_mode`: `"Off"`（学習日）または `"Shift"`（仕事の日）
  - `reflection`: 主観評価
    - `worked`: うまくいったこと（Fact+Why形式、最大3項目）
    - `slipped`: うまくいかなかったこと（Fact+Why形式、最大3項目）
    - `insight`: 気づき・学び（Fact+Why形式、最大3項目）
    - `strategy_for_next_day`: 翌日の戦略（テキスト）
  - `sessions`: 行動ログ（配列）
    - `start_time`, `end_time`, `duration_min`: セッションの時刻と時間
    - `category`: カテゴリ（Coding, Planning, Reading, Interview, Other）
    - `original_type`: 元のセッションタイプ
    - `deep_score`: 集中度（1.0-5.0、小数第1位）
    - `notes`: セッションメモ

#### 2. Daily Review v2フォーマット（元ファイル: `daily/daily_YYYY-MM-DD.md`）
各日のDaily Reviewファイルには以下のセクションが含まれます。週次分析では、これらを参照して詳細な引用を行います。

**■ Stats**（11フィールド）:
- Day Mode, Budget（予算学習時間）, Total Min, 純粋な学習時間, 休憩時間, 運動時間, Avg Deep Score, Top1（最優先目標）, Done条件（達成基準）, 切れたら→（代替タスク）

**■ Top1 / Done条件 達成度フィードバック**（AI生成）:
- 定量面: Top1関連セッション数・時間、Done条件の達成有無、達成度%
- 定性面: 達成困難だった原因、認知的ブロッカー、学習成果の評価

**■ 時間配分フィードバック**（AI生成）:
- 定量面: Budget達成率、時間帯別学習時間、非学習時間内訳
- 定性面: 学習リズム分析、DS時間帯別パターン、効率分析

**■ Worked / Slipped / Insights**（Fact+Why形式）:
- 各セクション最大3項目、`**Fact**: 事実` + `**Why**: 原因・メカニズム` のペア

**■ 今日の学習記録**:
- 今日やったこと、詰まったこと、なぜそうなったか、何を学んだか

**■ Study Strategy for Next Day**: 翌日の戦略

**■ Technical Learnings**（オプション）: Q&A・コード形式

※ 週次分析では、JSON `reflection`フィールドの情報と、Daily Review内のAI生成フィードバックを**両方**活用してください。日次レベルの分析を週単位に統合することで、より深い洞察が得られます。
```

### 3.3 分析フレームワーク

```markdown
【分析フレームワーク】

以下の3段階（Measure → Learn → Build/Pivot）で分析してください。

---

## 📊 1. Measure（事実確認）

事実ベースで1週間を俯瞰します。

### 1.1 時間の使い方
- **カテゴリ別時間**: `sessions`内の`category`別の合計時間と割合（Coding / Interview / Reading / Planning / Other）
- **Day Mode別学習時間**: `day_mode: "Off"`（学習日）と`day_mode: "Shift"`（仕事の日）の平均学習時間を比較

### 1.2 時間帯別分析（NEW）
- **午前ブロック**（07:00-12:00）: DS平均、Cognitive Headroom推定（DS≧4.0なら高Headroom）
- **午後ブロック**（12:00-17:00）: DS平均、昼休憩後の再開ラグ（分）
- **夜間ブロック**（17:00-22:00）: DS平均、Cognitive Headroom推定（DS≦3.0なら低Headroom）

※ Cognitive Headroomは直接計測されないが、DS水準とセッションnotesの「眠気」「疲弊」記述から推定

### 1.3 集中度（Zone Analysis）
- **Flow状態セッション**: `deep_score ≧ 4.0`のセッションリスト（日付、時間帯、category、duration_min、notes）
- **共通傾向**: Flow状態セッションに共通する条件（時間帯、カテゴリ、直前の行動、セッション継続時間など）

### 1.4 Top1達成トレンド（NEW）
各日の `■ Top1 / Done条件 達成度フィードバック` から、以下を集計:
- **Top1達成率**: 達成/未達の日数と割合
- **Done条件達成パターン**: 時間制約（12:00まで）を守れた日 vs 成果達成を優先して超過した日
- **未達成の主因**: AI生成フィードバックの定性面から、認知的ブロッカーや時間消費の根本原因を抽出

### 1.5 Budget達成トレンド（NEW）
各日の `■ 時間配分フィードバック` から、以下を抽出:
- **Budget達成率の推移**: 日別のBudget達成率（%）と平均
- **Budget超過の解釈**: Flow状態継続による肯定的超過 vs 計画甘さによる超過を区別
  - Flow継続: 超過日にDS≧4.0の長時間セッションがあり、notes に「切り替えたくない」「Flow継続」の記述
  - 計画甘さ: 超過日に設計フェーズ延長や見積もり乖離の記述
- **非学習時間ブロックのパターン**: 長時間休憩（≧60min）、運動後の再開遅延（≧30min）の頻度と発生タイミング

### 1.6 新メトリクスのトレンド（NEW）
- **Satisfaction**（満足度、1-5）: 各日のSatisfactionスコア（Daily Review内に記録がある場合）の推移
- **Focus Metric**（集中持続度）: 関連タスクを連続実行したセッション群の抽出（例: rails new → DB設計 → ER図で計200min）
- **Friction**（摩擦度、1-5）: セッションnotesに「言語化できず」「断念」の記述がある場合のFriction推定

---

## 💡 2. Learn（深層分析）

JSON内の `reflection`（`worked`, `slipped`, `insight`）、`sessions` ログ、Daily ReviewのAI生成フィードバックを相関させて分析します。

### 🚀 2.1 Good & Pattern（勝ちパターン）

「Deep Scoreが高かったセッション」や「Satisfactionが高かった日」に共通する条件・行動は何か？

#### Top Performer
- **最も生産性が高かった日**: `deep_score`平均が高い日とその理由
  - 根拠: その日の `reflection.worked` や `sessions.notes`、`■ Top1 / Done条件 達成度フィードバック`から引用

#### Success Factors
- **`reflection.worked`に記述された内容と実際の高スコアセッションの因果関係**を抽出
  - 例: 「明確なDone条件」→ 午前ブロックDS平均4.2維持
  - 例: 「昼食後即再開」→ 再開ラグ6分のみ

#### Flow状態の維持パターン（NEW）
- **Flow中の「切り替えない」判断の成功例**: DS 4.0以上が連続したセッション群の分析
  - 例: 「午後もrails new（DS=3.7、Focus=5）のFlowが続いた → 切り替えず継続 → Budget 142.5%達成」
  - If-Thenルール「Flow状態（DS≧4.0）なら切り替えない」の適用実績

#### 週次戦略experiment検証（NEW）
前週の戦略（`generate_weekly_strategy_from_chat.md`で生成された場合）の`experiment`フィールドを参照:
- **`experiment.action`が実行されたか**: セッションログとの突合
- **`experiment.hypothesis`が正しかったか**: 実績データで検証
- **`experiment.success_definition`を満たしたか**: 成功基準の達成度評価

### 🚧 2.2 Bottle Neck（阻害要因）

「計画通り進まなかった日」や「Deep Scoreが低かったセッション」のトリガーは何か？

#### Trigger
- **`reflection.slipped`や低スコア時の`sessions.notes`から、集中を阻害した具体的要因を特定**
  - 環境要因: 時間帯（夜間Headroom=2）、騒音、割り込み
  - 体調要因: 眠気、疲弊、飲酒後の翌朝
  - 感情要因: 焦り、不安、完璧主義バイアス
  - タスク要因: 難易度、見積もり乖離、技術的ハマり

#### Emotion & Cognition
- **焦り、眠気、技術的ハマりがDeep Scoreにどう影響したか**を分析
  - 例: 「30min超過の技術的ハマり → DS 2.7 → 『切れたら→』未移行で悪化」
  - 例: 「夜間のHeadroom=2 → まとめフェーズ失敗（Friction=5）」

#### 認知リソース管理の問題（NEW）
- **Cognitive Headroom低下時のタスク選択が適切だったか**
  - 「切れたら→」タスクへの移行タイミング: 集中低下を自己感知して移行できたか
  - 夜間ブロックでの高負荷タスク配置: 新概念学習やアウトプット型作業を夜間に配置していないか

---

## 🏗️ 3. Build / Pivot（来週の戦略）

「来週、何か1つだけ変えるとしたら？」という視点で提案します。

### 3.1 STOP（やめること）
データから判明した「コスト対効果が低い行動」や「繰り返されているSlippedの原因」

- 例: 「夜間のまとめフェーズ（Headroom=2でFriction=5）」→ STOP
- 例: 「設計フェーズの完璧主義（12:00タイムボックス無視 → Done条件未達）」→ STOP

### 3.2 START（始めること）
`reflection.insight`や`Good Pattern`から導き出される、来週試すべき具体的なアクション

- 例: 「まとめフェーズを午前ブロック（Headroom高）に移動」→ START
- 例: 「12:00強制打ち切りのIf-Thenルール（タイマー設定）」→ START

### 3.3 The One Thing（重点実験）（NEW）
来週試す**1つだけ**の改善を明示

**フォーマット**:
- **行動**: 具体的なアクション（実行可能、測定可能）
- **仮説**: なぜそれをやるのか、どうなるはずか
- **成功定義**: 何をもって成功とみなすか（定量基準）

**例**:
- 行動: 朝の計画で「Top1」と「切れたら→」の2行だけ書く。時間ではなく集中低下の自己感知をトリガーに切り替え
- 仮説: 「切れたら→」の事前設定により、集中低下時の移行コストが下がり、午後のリスタートが円滑化される
- 成功定義: Off日の過半数（3日以上/5日中）で「切れたら→」への移行が実行され、移行後セッションのDSが3.0以上を維持

### 3.4 Risk Management（想定リスクとIf-Thenプラン）（NEW）
来週想定されるリスクと、事前に打っておく対策

**フォーマット**:
- If: リスク条件（トリガー）
- Then: 対策アクション

**例**:
- If: 同じエラー/問題に30min以上進展がなければ → Then: 問題を1行で言語化して「切れたら→」タスクに移行
- If: Shift帰宅後に「何をするか決まっていない」状態なら → Then: 出勤前の朝5minで「帰宅後の1タスク」を決定
- If: 実装に没頭してDS 4.0以上が続いているなら → Then: 切り替えない（Flow状態を尊重）

### 3.5 Success Criteria（来週の成功基準）（NEW）
来週全体の成功を判断する基準

**例**:
- Off日DS平均 3.3以上を維持
- ミニアプリ基本構造（rails new → DB設計 → 主要画面のCRUD）が動作状態
- 「切れたら→」移行を3回以上実行し、移行後DSが3.0以上

### 3.6 Strategy連続性チェック
各日の`reflection.strategy_for_next_day`が翌日に実行されたか、`sessions.notes`と突合して追跡

- **未実行の戦略**: 翌日のセッションログに該当記述がない戦略をリストアップ
- **阻害要因分析**: 未実行の原因（Shift Dayで時間不足、優先度変更、忘却など）を推定
- **来週の改善**: 戦略実行率を上げるための仕組み（例: 朝の計画で前日戦略を確認）を提案
```

### 3.4 出力ルール

```markdown
【出力ルール】

#### 1. 口調
- **客観的かつコーチトーン**: データから導かれる気づきを提示する姿勢
- 「〜のようです」「〜が見受けられます」「〜という傾向があります」など、断定を避けた表現
- ユーザーの自律的な改善を促す問いかけ（「〜は適切でしたか？」「〜を試す価値がありそうです」）

#### 2. 引用形式（5つのルール）
1. **日付参照**: 根拠となる日付を必ず`[YYYY-MM-DD]`の形式で添える
2. **セッション引用**: セッションnotesから引用する場合、`「〜」[YYYY-MM-DD, HH:MM-HH:MM, category]`の形式
3. **Daily Review引用**: Worked/Slipped/Insightsから引用する場合、`「〜」[YYYY-MM-DD, Worked/Slipped/Insights]`の形式
4. **AI生成フィードバック引用**: Top1達成度や時間配分フィードバックから引用する場合、`「〜」[YYYY-MM-DD, Top1達成度/時間配分]`の形式
5. **weekly_strategy_context引用**: 前週の戦略JSONから引用する場合、`experiment.action: 「〜」`のように明示

#### 3. 出力長さ
- **全体**: 2500-3500文字（A4用紙2-3枚相当）
- **Measure**: 800-1200文字（事実を簡潔に）
- **Learn**: 600-1000文字（パターンとボトルネックを深掘り）
- **Build/Pivot**: 400-600文字（アクション提案を具体的に）
- スマホでスクロールして読める分量を意識し、箇条書き中心で構成

#### 4. 構造
- 各セクションは絵文字（📊, 💡, 🏗️）で視覚的に区別
- 副見出し（###）で項目を分割し、一覧性を確保
- 長い文章は避け、1項目2-3行以内に収める
```
