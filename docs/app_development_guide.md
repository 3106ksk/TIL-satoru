## 1. 開発前のマインドセット（心構え）

技術選定やコーディングに入る前に、

### 「就活のための技術」ではなく「ユーザーのための技術」
*   「就活に有利だから」という理由だけで技術を選ばないこと。
*   ユーザーの課題を解決するために技術を使い倒す姿勢（泥臭い実装や工夫）こそが、結果的にエンジニアとしての評価に繋がります。

### 主体性（Take Ownership）を持つ
*   AIや講師に「正解」を求めるだけの「プロンプト入力者」にならないこと。
*   「自分はこう考えるが、どう思うか？」という仮説を持って相談し、AIが出力したコードも意味を理解して責任を持って使う姿勢が重要です。

### 期間の意識
*   ミニアプリであれば2週間程度を目安にします。
*   1ヶ月以上かけるのはスコープが広すぎます。

## 2. 準備段階で言語化・決定すべきこと

コードを書き始める前に、以下のプロセスを経て企画を固めます。

### ① 課題（Why）とペルソナ（Who）の深掘り
「作りたいものを作る」だけでは誰にも使われません。

*   **課題の分解**:
    *   「運動が続かない」などの漠然とした悩みではなく、「面倒だから？」「成果が見えないから？」「孤独だから？」と理由を細分化し、どのポイントを解決するか絞り込みます。
*   **ペルソナ設定**:
    *   「誰の、どんな瞬間」を助けるアプリなのかを具体的に定義します。

### ② 競合調査と差別化（What）
*   **既存アプリを知る**:
    *   すでにあるアプリを知らずに作ると説得力が下がります。競合を調査した上で、「既存のものと何が違うのか？」を説明できるようにします。
*   **差別化の定義**:
    *   奇抜な機能を入れることが差別化ではありません。「このアプリは、ユーザーのこの瞬間を一番助けられる」という体験の質や解決のアプローチで差別化します。

### ③ ユーザビリティと体験設計（How）
*   **入り口は極力シンプルに**:
    *   ユーザーは初期設定や入力項目が多いと離脱します。例えば、「まずはつぶやくだけ」のように、操作回数を減らし、直感的に使える設計（入り口を優しく）にします。
*   **+1 成功状態の定義**:
    *   ユーザーが最終的にどういう状態になればハッピーなのかを決め、そこから逆算して必要な機能を考えます。

### ④ MVPスコープの決定（削ぎ落とし）
「あったらいいな」は削ぎ落とし、コア機能に集中します。

*   **必須機能**:
    *   入力、計算、表示など、課題解決に直結する機能のみ。
*   **後回しにする機能**:
    *   凝ったUI、PDF出力、カレンダー連携、SNSシェアなどの外部連携は、工数が膨らむため初期MVPには含めない判断が賢明です。

### ⑤ 懸念点の洗い出し（リスク管理）
機能を出した後に、ネガティブな要素をリストアップします。

*   **技術的懸念**:
    *   実装難易度が高い箇所はどこか？
*   **ユーザー行動の懸念**:
    *   「面倒くさがりの人は、そもそもアプリを開かないのでは？」といった根本的なリスクに対し、どう対策（バッジ機能、可視化など）するかを考えます。

### ⑥ 技術選定と調査
*   **ドメインとの整合性**:
    *   自分が進みたい業界（例：ヘルステック、ゲーム、金融）で使われている技術やトレンド（ドメイン駆動設計など）を意識して選定すると、キャリアとの一貫性が生まれます。
*   **+1 調査アプローチ**:
    *   ChatGPT等で「実装ロードマップ」を作成し、出てきたキーワードを公式ドキュメントや技術記事で裏取りします。

## まとめ：開発着手前のチェックリスト

準備が整っているか、以下のリストで確認してみてください。

| 項目 | 確認内容 |
| :--- | :--- |
| **課題の特定** | 「なんとなく」ではなく、具体的な痛みを特定できているか？ |
| **ターゲット** | 「誰の」課題か、一人の人間が想像できるレベルまで落ちているか？ |
| **競合優位性** | 既存アプリとの違いを、機能数ではなく「体験」で説明できるか？ |
| **スコープ** | 2週間で作れる規模か？（外部連携や凝ったUIをMVPに入れていないか？） |
| **UX設計** | ユーザーの入力負荷（面倒くささ）を極限まで減らせているか？ |
| **技術選定** | その技術を選ぶ「理由（必然性や学習目的）」を語れるか？ |
| **懸念点** | 「使われないリスク」や「実装の壁」を事前にリストアップできているか？ |

まずはこれらの項目をテキスト（企画書やメモ）に落とし込み、**「これでユーザーの課題は本当に解決するか？」**を自問してから実装に入ると、手戻りが少なく、質の高いアプリになります。
