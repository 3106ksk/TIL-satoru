## Technical Learnings

### Q：セッションとは

ブラウザからアプリへのリクエスト時に使われるPost、GetなどのHTTPプロトコルは情報を保持しない。つまりステートレスな状態。

反対にセッションを使うことはステートフルな状態に変えること。

ステートレス：情報を保持しない

ステートフル：情報を保持　例) セッション管理

セッション構成要素：「値のハッシュ」 と　「セッションID」

セッションIDは、ハッシュを特定するために使用される

Railsでは、上記のステートフルな状態を実現するために下記のsessionメソッドを使用して
値の保存や取り出しが行われている

```ruby
session[:user_id] = @current_user.id
User.find(session[:user_id])
```

参照：Rails セキュリティガイド | 2.1セッションとは何か？

#### 具体例

```
1. ログイン時
   session[:user_id] = 1 を実行
   ↓
   Railsが { user_id: 1 } を暗号化・署名
   ↓
   暗号化されたデータをCookieとしてブラウザに送信
   ↓
   ブラウザがCookieを保存

2. 次回リクエスト時
   ブラウザがCookieをサーバーに送信
   ↓
   Railsが署名を検証（改ざんされていないか確認）
   ↓
   復号化して { user_id: 1 } を取得
   ↓
   session[:user_id] で値にアクセス可能に
```

ブラウザに保存されているCookieには常にセッションIDがある。つまり、ブラウザからのリクエストには毎回cookiesを送信している。（ステートフルな処理）

#### 解釈

だからカリキュラムで実装した以下のコード処理でログインを省くことができていた

- idカラムがsession[:user_id]の値を持つユーザーレコードをテーブルから探している

```ruby
def current_user
  @current_user ||= User.find_by(id: session[:user_id]) if session[:user_id]
end
```

また公式を参照すると以下のような書き方が記載してあり、現状はセッションを使う時は以下のように書くと理解することにした。理解としては、idをセッションに保存している意図はidでログインユーザーを判別するためであると理解している。
なので質問３はそもそも設計としてemailだと保守性が弱いと考える

```ruby
session[:user_id] = @current_user.id
User.find(session[:user_id])
```

### Q：もし `session[:user_id]` ではなく `session[:email]` にメールアドレスを保存する設計だったら、何か問題があるか？

解答：保守性が下がる

**idを使う理由**

1. **一意性が保証されている**：DBのプライマリーキーなので絶対に重複しない
2. **不変である**：ユーザーがメールアドレスを変更しても `id` は変わらない
3. **データサイズが小さい**：整数なので軽量（Cookieの容量制限対策）
4. **検索が高速**：プライマリーキーなのでインデックスが効いている
5. **セキュリティ**：メールアドレスより推測されにくい

**emailを保存した場合の問題**

- ユーザーがメールアドレスを変更したらセッションが無効になる
- メールアドレスは文字列なので容量が大きい
- 検索速度が `id` より遅い可能性がある
