## 2. 技術面談の本質的価値の再定義

### 2-1. RUNTEQエンジニアスキルとのマッピング

RUNTEQが定義する5つのエンジニアスキル:
1. 計画性
2. 素直さ
3. コミュニケーション力
4. 自主性
5. 論理的思考

#### 技術面談が鍛えるスキル

| スキル | 技術面談の効果 | 根拠 |
|---|---|---|
| **3. コミュニケーション力** | ✅ **直接的** | 「自分の情報や考えを整理し、相手に伝わるよう工夫している」 |
| **5. 論理的思考** | ✅ **直接的** | 「面談や相談の場で、『前提→根拠→結論』の順で話を構成」 |
| **2. 素直さ** | △ 間接的 | 「アドバイスを受けたあと、自分が次に何をするかを言語化」 |
| **1. 計画性** | ❌ 効果薄い | 逆算型計画は週次戦略設計で鍛える |
| **4. 自主性** | ⚠️ **リスクあり** | 「分からないことを自分で調べず、すぐに周囲の人やAIに聞いてしまっている」 |

#### 重要な発見: 技術面談の二面性

技術面談は**諸刃の剣**:
- ✅ **強化**: コミュニケーション力、論理的思考
- ⚠️ **阻害リスク**: 自主性(答えをもらう癖がつく)

### 2-2. 技術面談のROI方程式

```
技術面談の効果 = 言語化能力 × トライアンドエラーの蓄積
```

**言語化能力がゼロなら、どれだけ技術面談をしても効果はゼロ。**

### 2-3. 良い相談 vs 悪い相談

#### 悪い相談パターン(RUNTEQスキル違反)

```
学生: 「エラーが出ました。どうすればいいですか?」
講師: 「何を試しましたか?」
学生: 「何も...」
→ 講師は答えを教えるだけ (学生の成長なし)
```

#### 良い相談パターン(RUNTEQスキル準拠)

```
学生: 「このエラーが出ました。
       仮説1: ルーティングの問題 → 試したこと → 結果NG
       仮説2: データベースの問題 → 試したこと → 結果NG
       仮説3: XXXが原因? → これは未検証です。どう思いますか?」
講師: 「仮説3は良い視点。加えてYYYも確認してみて」
→ 学生の思考プロセスが鍛えられる
```

**後者は5つのRUNTEQスキル全てを発揮している。**

### 2-4. 技術面談の新定義

**Before (今回のOBL)**:
- 目的: 回数を増やす(量)
- 手段: とにかく予約する
- 結果: 答えをもらう場

**After (次の6週間)**:
- 目的: **思考を外部化して整理する場**(質)
- 手段: 3段階プロセス(準備→実施→復習)
- 結果: 5つのRUNTEQスキルを統合的に鍛える

## 3. 言語化能力の重要性

### 3-1. 言語化能力が技術面談の前提条件

**重要な洞察**:
> 「設計意図やコードのロジックを完璧でなくても言語化できる」ことが前提となる。その意図で実装をしたりリサーチをしてエラーが出てトライアンドエラーをしてできなかったときに、講師とのやりとりをすることで真に効力を発揮する。

### 3-2. 言語化能力の3層構造

| Layer | 内容 | 源泉 |
|---|---|---|
| **Layer 1: 語彙** | 専門用語、概念、パターン名 | 技術書籍、公式ドキュメント、技術記事 |
| **Layer 2: 構造** | アーキテクチャ、設計パターン、トレードオフ | 技術書籍(体系的)、実装(実体験) |
| **Layer 3: 文脈** | 「なぜこの設計か」「なぜこのgemか」 | 実装の繰り返し、トライアンドエラー、技術面談 |

**Layer 3の言語化には Layer 1 + Layer 2 が必須。**

### 3-3. 言語化能力の段階的習得プラン

#### Week 1-2: Level 1 (不完全でもOK)
- **目標**: エラーの状況を最低限説明できる
- **訓練**: Daily Reviewで「何をした→何が起きた」を記録
- **技術面談**: 不完全でも講師に伝える練習

#### Week 3-4: Level 2 (構造化)
- **目標**: トライアンドエラーの過程を整理できる
- **訓練**: Daily Reviewで「試したこと→結果→次の仮説」を記録
- **技術面談**: 「前提→根拠→結論」で説明

#### Week 5-6: Level 3 (メタ認知)
- **目標**: 設計意図やロジックの理由を説明できる
- **訓練**: Daily Reviewで「なぜこの設計にしたか」を記録
- **技術面談**: 「なぜ」を含めた深い議論

### 3-4. Daily Review × 技術面談の相乗効果

| Daily Reviewでの訓練 | 技術面談での発揮 |
|---|---|
| Fact: 何をした | 前提: 実装した内容 |
| Why: なぜそうした | 根拠: 設計意図 |
| Worked/Slipped | トライアンドエラーの過程 |
| Insight | 仮説・疑問点 |
| Next | 次に何を確認したいか |

**Daily Review = 言語化能力のトレーニング場**
**技術面談 = 言語化能力の実践場**

## 4. インプット×アウトプット循環の設計

### 4-1. 現状の問題: 語彙なき実装の悪循環

#### Reading時間の推移

| Week | Reading時間 | 割合 | 内容 |
|---|---|---|---|
| Week 5 | 13min | **1%** | 眠気で早期終了 |
| Week 6 | 43min | **3.2%** | 「オブジェクト指向でなぜ作るのか」 |

**これは極端に少ない。**

#### 記事執筆の不安定性

| Week | 記事執筆 | 結果 |
|---|---|---|
| Week 5 | あり | DS 4.0×2 (高品質) |
| Week 6 | **ゼロ** | 「記事執筆並行」未着手 |

#### 問題の構造

```
実装(豊富) → エラー・課題発生
     ↓
語彙不足 → 「なぜ?」を言語化できない
     ↓
技術面談で相談 → 表面的な解決(答えをもらうだけ)
     ↓
次の実装でまた詰まる
```

**これは RUNTEQスキル「自主性」のBadアクションそのもの:**
> 「分からないことを自分で調べず、すぐに周囲の人やAIに聞いてしまっている」

### 4-2. 理想形: インプット→実装→言語化→記事執筆の循環

```
技術書籍・記事(インプット) → 語彙・構造の習得
     ↓
実装で試す → 概念を体験に変換
     ↓
エラー発生 → 「なぜ?」を語彙を使って考える
     ↓
技術面談で議論 → 深い理解(単なる答えではない)
     ↓
記事執筆 → 理解の定着・言語化の完成
     ↓
次の実装で応用できる (再利用可能な知識)
```

**これが RUNTEQスキル「自主性」のGoodアクション:**
> 「疑問が生じた際、複数の手段（公式ドキュメント、書籍、動画など）を参照し、情報を収集している」

### 5-3. 技術面談の3段階プロセス

**頻度**: 週1回30分 (量から質へ)

#### Phase 1: 準備 (Prep, 15-20min)

**目的**: 論理的思考の訓練

**テンプレート**:
```markdown
## 技術面談Prep

### 相談したいこと
[1文で結論]

### 前提
- 実装内容: XXX
- 目的: YYY
- 環境: ZZZ

### トライアンドエラーの記録
1. 試したこと: AAA → 結果: BBB
2. 試したこと: CCC → 結果: DDD
3. 試したこと: EEE → 結果: FFF

### 現在の仮説
- 仮説1: XXXが原因? (自信度: 30%)
- 仮説2: YYYが原因? (自信度: 50%)

### 聞きたいこと (3つ以内)
1. 仮説2は妥当か?
2. 他に確認すべき箇所は?
3. この設計の改善点は?
```

**測定**: Daily ReviewのPrepセッションにDS記録

#### Phase 2: 実施 (Interview, 20-30min)

**目的**: コミュニケーション力の訓練

**チェックリスト**:
- [ ] カメラオンで参加
- [ ] 相槌・表情でリアクション
- [ ] 構造化された説明 (結論→前提→根拠)
- [ ] アドバイスを受けたら「次に何をするか」を言語化

**測定**: Sessions DBにInterview記録 + Notes欄に「次のアクション」

#### Phase 3: 復習 (WrapUp, 10-15min)

**目的**: 素直さの訓練

**チェックリスト**:
- [ ] 面談で得た学びを3点にまとめる
- [ ] 24時間以内の実践タスクを決定
- [ ] Daily Reviewの「Insight」に記録

**測定**: 翌日Daily Reviewで実践を確認

### 5-4. 自主性とのバランス設計

**ルール**: 「30分自己解決 → それでも詰まったら面談予約」

| 状況 | 行動 | 鍛えられるスキル |
|---|---|---|
| エラー発生 | まず自分で30分調査 | 自主性 |
| 30分で解決せず | 調査結果を整理して面談予約 | 論理的思考 |
| 面談で相談 | 「前提→根拠→結論」で説明 | コミュニケーション力 |
| アドバイス受領 | 次のアクションを言語化 | 素直さ |
| 24時間以内 | 実践してDaily Reviewに記録 | 計画性 |

**この流れで、RUNTEQの5つのスキル全てが鍛えられる。**

## 7. リスクと対策

### 7-1. リスク1: Reading時間の確保失敗

**兆候**:
- Shift Day帰宅後にReading予定 → 実施されず
- 「切れたら→」タスクに設定 → 発動ゼロ

**対策**:
- If-Then: 「Shift Day帰宅したら、まず30分Reading → その後ミニアプリ」
- 週次レビューで Reading実績を確認
- Week 2でReading 1h未満なら、Week 3で強制Reading Day設定

### 7-2. リスク2: 記事執筆の先送り

**兆候**:
- Week 6の再現: 「記事執筆並行」未着手
- 「ミニアプリ開発が忙しい」という理由で後回し

**対策**:
- If-Then: 「日曜午後は必ず記事執筆。ミニアプリは金曜までに区切る」
- 記事執筆は「週テーマ」に入れない (スコープ過大を避ける)
- 記事執筆を「切れたら→」タスクとして配置 (中負荷タスク)

### 7-3. リスク3: 技術面談の準備不足

**兆候**:
- Prep時間を取らず、いきなり面談予約
- 「何を聞くか」が不明確なまま参加

**対策**:
- If-Then: 「技術面談を予約したら、必ずPrepセッションを計画に入れる」
- Prepなしの面談は禁止ルール
- Daily ReviewでPrep実施を確認

### 7-4. リスク4: デプロイ作業でハマる

**兆候**:
- Week 5でデプロイ着手 → 想定外のエラーで3日消費

**対策**:
- Week 4でデプロイ手順書を事前作成 (30min)
- デプロイは複数回試す (開発環境で練習)

### 7-5. リスク5: ミニマムスコープ超過

**兆候**:

- MVP定義を守れず、Week 4で実装未完了

- Week 1でMVP定義を明文化 (必須機能3つのみ)
- 追加機能は「次サイクル候補」リストへ
- 週次戦略でスコープ確認

### 7-6. リスク6: 完璧主義の再発

**兆候**:
- 設計フェーズで12:00タイムボックス無視
- ER図・画面遷移図を「完璧に」仕上げようとする

**対策**:
- 12:00アラーム設定 + 強制切り替えルール
- Done条件に「完璧でなくてもOK」を明記
- 「8割で次へ」を合言葉に

## 10. 結論

### 10-1. 前回OBLの総括

技術面談20回という目標は**適切**だったが、**実行メカニズム**(週次統合、トリガー設計、質の設計)が欠けていた。未達は「実行力の欠如」ではなく、**戦略的PIVOTの結果**であり、失敗ではない。

6週間で得た最大の資産は:
- 再現可能な成功パターン5つ
- 阻害要因4つの特定
- **技術面談の本質的価値の理解**: 答えをもらう場ではなく、思考を整理する場

### 10-2. 技術面談の再定義

技術面談の効果 = **言語化能力** × トライアンドエラーの蓄積 × 講師の質

言語化能力の源泉 = **インプット(技術書籍・記事) × アウトプット(実装) の循環**

現状の問題:
- Reading 2% (語彙不足)
- 記事執筆不安定 (理解の定着不足)
- **語彙なき実装の悪循環**

### 10-3. 次OBLへの統合戦略

**OBL**: ミニアプリ完成 + デプロイ
**アプローチ**: Just-in-Time Learning

**黄金比**:
- Reading: 10-15% (週2-3h)
- 実装: 60-65% (週14-16h)
- 記事執筆: 10% (週2-3h)
- 技術面談: 3-5% (週1回30min)
- Planning: 10-15% (週2-3h)

**循環構造**:
```
実装前: Reading (語彙・構造の習得)
  ↓
実装: ミニアプリ開発 (体験への変換)
  ↓
Daily Review: 「なぜ」を言語化 (語彙を使う)
  ↓
技術面談: 深い議論 (週1回、3段階プロセス)
  ↓
記事執筆: 理解の定着 (週1本、定着率90%)
  ↓
次の実装で応用 (再利用可能な知識)
```

### 10-4. 成功の定義

6週間後、以下が達成されていれば成功:

1. **OBL達成**: ミニアプリがデプロイされ、ユーザーフィードバック3件取得
2. **プロセス改善**: Reading 10-15%、記事執筆週1本、技術面談活用率100%
3. **スキル向上**: RUNTEQの5つのスキル全てが鍛えられている
4. **資産化**: 技術記事6本、再利用可能な知識の蓄積
5. **習慣定着**: Reading・記事執筆のトリガーが自動化

### 10-5. 哲学的結論

**語彙なき実装は、地図なき航海だ。**

- **Reading**: 地図を手に入れる
- **実装**: 航海する
- **トライアンドエラー**: 嵐に遭う
- **技術面談**: ベテラン船長に相談する (地図があるから会話が深い)
- **記事執筆**: 航海日誌を書く (次の航海で使える)

Reading 2%では、地図なしで航海している。だから毎回迷う。

次の6週間では、**Just-in-Time Learningで、インプット×アウトプット×言語化の循環を回し、再利用可能な知識を蓄積する。**

技術面談は「答えをもらう場」ではなく、**「思考を外部化して整理する場」**だ。
