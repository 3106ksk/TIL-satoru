# Reading Trigger Design - OBL Cycle 2 (2026-02-23 〜 2026-04-05)

**作成日**: 2026-02-23
**対象期間**: OBL Cycle 2（ミニアプリ完成＋デプロイ）
**目的**: Reading時間の確保とトリガー設計を具体化し、語彙なき実装の悪循環を脱却する

---

## 背景と課題

### 過去の実績

| Week | 学習時間 | Reading時間 | Reading割合 |
|---|---|---|---|
| Week 5 (2/9-2/15) | 24.4h | 13min | 1% |
| Week 6 (2/16-2/22) | 22.5h | 43min | 3.2% |

### 構造的問題

1. **語彙なき実装の悪循環**:
   ```
   実装 → エラー → 語彙不足で言語化できない →
   技術面談で答えをもらうだけ → 次でまた詰まる
   ```

2. **トリガー設計の失敗パターン**:
   - 自己感知トリガー（「切れたら→」）は発動率ゼロ
   - Shift Day帰宅後の計画未設定 → 空白時間発生
   - 夜間高負荷タスク → 失敗率100%

3. **技術面談の効果**:
   ```
   技術面談の効果 = 言語化能力 × トライアンドエラー × 講師の質
   ```
   - 言語化能力がゼロなら、どれだけ技術面談をしても効果はゼロ
   - 言語化能力の源泉 = インプット（技術書籍・記事）× アウトプット（実装）

## Reading時間の目標設定

### 次OBLサイクルの目標

**週3h（12-13%）** - 過去実績の5倍増

| 指標 | 現状（Week 5-6平均） | 目標 | 増加率 |
|---|---|---|---|
| 週次Reading時間 | 28min | 180min | 約6.4倍 |
| 学習時間に占める割合 | 2% | 12-13% | 6倍 |

### 週3h = 180min/週の配分

**配分案A: Off Day朝集中型**（採用）

| タイミング | トリガー | 時間 | 週回数 | 週合計 |
|---|---|---|---|---|
| **Off Day朝（9:00-9:30）** | 朝食後 → 即Reading | 30min | 5日 | 150min |
| **Shift Day帰宅後** | 帰宅 → 着替え → 即Reading | 15min | 2日 | 30min |
| **合計** | - | - | - | **180min** |

**選定理由**:
- シンプルで習慣化しやすい
- 朝の固定枠は成功率が高い（Cognitive Headroom 4-5）
- トリガーが明確（朝食後、着替え後）
- 複雑なトリガー設計は過去に失敗している

## if-thenプランニング（具体的トリガー）

### トリガー1: Off Day朝Reading（週150min）

#### if-thenプラン

```
もし「朝食を終えた」なら、その時「9:00-9:30の30minで、今日の実装テーマに関連するドキュメント/記事を読む」
```

#### 具体的プロセス

**1. 前夜準備（23:00、Daily Planning時）**:
- [ ] 明日の実装テーマを特定（例: DB設計、CRUD実装、検索機能）
- [ ] 読むべきドキュメント/記事のURLをDaily Planに明記
- [ ] デバイスのブックマークに追加
- [ ] スマホのリマインダーに「9:00 Reading」を設定

**2. 朝食後（9:00）**:
- [ ] コーヒーを入れる
- [ ] ブックマークを開く
- [ ] タイマー30min設定
- [ ] **読むだけ**（ノート取りは後回し、まず語彙を浴びる）

**3. 9:30終了時**:
- [ ] Sessions DBに記録（Reading Type, 30min, DS評価）
- [ ] Daily Reviewの「Fact」に「〇〇について読んだ」を1行記録
- [ ] 実装セッション開始（9:30-12:00）

#### Cognitive Headroom適合性

- **朝9:00**: Headroom 4-5（高い時間帯）
- **焦点**: 「深い理解」ではなく「語彙のインプット」
- **理解の深化**: 実装中に深まる（Just-in-Time Learning）

---

### トリガー2: Shift Day帰宅後Reading（週30min）

#### if-thenプラン

```
もし「Shift Dayから帰宅して着替えた」なら、その時「15minで翌日の実装テーマのドキュメントを読む」
```

#### 具体的プロセス

**1. 出勤前朝5min**（Bottleneck 1対策）:
- [ ] 帰宅後の1タスクを決定：「翌日の実装テーマ〇〇のドキュメント15min」
- [ ] スマホのリマインダーに設定
- [ ] Reading素材（URL/ページ番号）をメモ

**2. 帰宅後（18:30-19:00）**:
- [ ] 着替え
- [ ] リマインダー確認
- [ ] タイマー15min設定
- [ ] **読むだけ**（ノート取り不要、翌日の予習として）

**3. 15min終了時**:
- [ ] Sessions DBに記録
- [ ] 翌日のDaily Planに「今日読んだ内容を実装で試す」をメモ

#### Cognitive Headroom適合性

- **帰宅後18:30-19:00**: Headroom 2-3（低め）
- **負荷**: 15minの軽負荷Reading（予習レベル）
- **目的**: 深い理解は求めず、翌日の実装の「地図」を手に入れる

## Just-in-Time Learningとの統合

### Reading → 実装 → 言語化の循環

```
前夜（23:00）: Daily Planで明日の実装テーマを特定 + Reading素材をブックマーク
  ↓
朝（9:00-9:30）: 30min Reading（語彙・構造のインプット）
  ↓
午前（9:30-12:00）: 実装セッション（読んだ内容を試す）
  ↓
午後（13:00-18:00）: 実装継続（エラー発生 → 30min自己解決）
  ↓
夕方（18:00-18:15）: Daily Reviewの「Fact + Why」記録（語彙を使って言語化）
  ↓
週末（日曜午後）: 記事執筆（理解の定着、定着率90%）
```

### Reading内容の選定ルール

| Week | 週次マイルストーン | Off Day朝30min Reading | Shift Day帰宅後15min Reading |
|---|---|---|---|
| **Week 1** | DB設計 + rails new | Railsガイド「Active Record Migration」、データベース設計本の関連章 | 翌日のDB設計関連（正規化、インデックス、外部キー制約） |
| **Week 2** | CRUD実装完了 | Railsガイド「Active Record の基礎」、コントローラー/ビュー | 翌日のCRUD実装（バリデーション、エラーハンドリング） |
| **Week 3** | 検索機能実装 | Ransack gemドキュメント、検索機能の技術記事（Qiita） | 翌日の検索機能（クエリ最適化、UI/UX） |
| **Week 4** | デバッグ+UI改善 | デバッグ技術記事、Bootstrapドキュメント | 翌日のデバッグ/UI改善（Chrome DevTools、CSS） |
| **Week 5** | デプロイ | Heroku/Renderドキュメント、デプロイ手順記事 | 翌日のデプロイ作業（環境変数、DB移行） |
| **Week 6** | フィードバック収集 | 次サイクルの技術トピック探索 | 翌日のフィードバック対応（軽微な修正） |

### デフォルトReading素材（選定に迷った場合）

1. **最優先**: Railsガイド（公式、信頼性最高）
2. **次点**: 使用gemの公式ドキュメント/README
3. **補足**: 技術記事（Qiita、Zenn）- ただし公開日2024年以降

**選定時間制限**: 5min以内（ググり過ぎを防ぐ）

## リスクと対策

### リスク1: Off Day朝のReading未実施（最大リスク）

**兆候**:
- 朝食後にそのまま実装セッション開始
- 「今日は時間がない」という理由でスキップ

**if-then対策**:
```
もし「朝食後にReadingをスキップしたくなった」なら、その時「5minだけ読む。5min後にやめてもOK」
```

**根拠**:
- 最小限ルール（5min）で着手障壁を下げる
- 実際は5min読めば30min継続する可能性が高い（着手の力学）
- OBL_sprint.md Part 2 Step 7「小さな勝利を積む」の原則

**測定**:
- Daily Reviewで「朝食後→即Reading」発動を記録
- 週次レビューで発動率を確認（目標: 100%）

---

### リスク2: Shift Day帰宅後の計画忘れ（中リスク）

**兆候**:
- 出勤前朝5minで計画を立てなかった
- 帰宅後にリマインダーを無視

**if-then対策**:
```
もし「出勤前朝5minで計画を立て忘れた」なら、その時「通勤中にスマホで翌日の実装テーマを決定し、Readingリンクを保存」
```

**根拠**:
- 電車内でも計画可能（スマホで完結）
- スマホのリマインダー+ブックマークで二重保険
- 教訓4「Shift Dayは別戦略が必要」の実践

**測定**:
- Shift Day出勤前の計画設定率（目標: 100%）
- Shift Day帰宅後Reading発動率（目標: 100%）

---

### リスク3: Reading素材の選定に時間を取られる（中リスク）

**兆候**:
- 9:00になっても「何を読むか」が決まっていない
- ググる時間で10min消費
- 30minのうち実際に読んだのは20min

**if-then対策**:
```
もし「前夜にReading素材を決めていなかった」なら、その時「Railsガイドの目次から今日の実装に最も近い章を選ぶ（5min以内）」
```

**根拠**:
- デフォルトはRailsガイド（常に信頼できるソース）
- 選定時間を5minに制限（完璧主義の回避）
- 「読まないよりマシ」の原則

**測定**:
- 前夜のReading素材決定率（目標: 80%以上）
- Reading開始時刻（目標: 9:05以前）

---

### リスク4: 実装での活用率が低い（中リスク）

**兆候**:
- 朝読んだ内容を実装で使わない
- 「読んだけど関係なかった」という感想

**if-then対策**:
```
もし「朝のReadingが実装で活用されなかった」なら、その時「Daily Reviewで『なぜ関係なかったか』を分析し、翌日のReading選定基準を修正」
```

**根拠**:
- Reading素材の選定基準が実装とズレている可能性
- PDCAサイクルで選定精度を向上
- 週次レビューで選定基準を改善

**測定**:
- Daily Reviewで「読んだ内容を試した」記録率（目標: 80%以上）
- 週次レビューで活用率を確認

## PIVOT基準（Week 3中間レビュー）

Week 3終了時点で以下の場合、トリガー設計をPIVOT:

| 指標 | GO基準 | PIVOT基準 | PIVOT対策 |
|---|---|---|---|
| **Off Day朝Reading実施率** | ≥80% (12/15日) | 60-79% (9-11/15日) | トリガーを「朝食後」から「起床後30min以内」に変更 |
| **週次Reading合計時間** | ≥5h (3週合計) | 3-4.9h (3週合計) | 目標を週2hに下方修正（段階的アプローチ） |
| **実装での活用率** | ≥80% | 50-79% | Reading内容の選定基準を見直し（より実装直結型に） |
| **トリガー発動率** | ≥90% | 60-89% | if-thenプランの文言を修正、リマインダー強化 |

### KILL基準（戦略的撤退）

以下の場合、Reading戦略を根本的に見直す:
- Off Day朝Reading実施率 < 60%（9/15日未満）
- 週次Reading合計時間 < 3h（3週合計）
- 実装での活用率 < 50%

**KILL時の対策**:
- 配分案Bまたは配分案Cへの切り替え
- 目標を週1.5h（6-7%）に下方修正
- Just-in-Time特化型への完全移行

## 習慣定着度の評価

### 自動化レベル（OBL Part 3 Step 7）

6週間後に以下の基準で自己採点:

| レベル | 状態 | 判定基準 |
|---|---|---|
| **5点** | 完全自動化 | リマインダー不要、朝食後に自然とReadingを開始している |
| **4点** | ほぼ自動化 | リマインダーを見て即発動、迷いなし |
| **3点** | 意識的実行 | リマインダー必須、時々スキップしそうになる |
| **2点** | 不安定 | 実施率60%以下、トリガーが機能していない |
| **1点** | 未定着 | 実施率30%以下、習慣化失敗 |

**目標**: Week 6終了時点で**レベル4以上**

**レベル3以下の場合**:
- 次サイクル（12週間目標）で習慣定着フェーズを延長
- トリガー設計を再設計
- 教訓3「習慣定着には6週間では不足」を再確認
